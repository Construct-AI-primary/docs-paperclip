# LOGISTICS-001: 01700 Design and implement shared logistics database schema

## Executive Summary

This issue creates the core database foundation for the 01700-logistics discipline. Every logistics feature — ContainerTracking, CustomsManagement, DailyDelivery, DocumentGeneration, InternationalShipping, SitePlanning, TrafficManagement, and MaterialFlow — depends on this schema. The design follows PAGE-KNOWLEDGE §1.1–1.2 architecture: core `logistics_records` table with UUID PK, RLS policies, composite indexes, monthly partitioning, Redis caching, and complete audit trail.

## Problem Statement / Scope

The logistics page relies on shared database entities that span all 8 domain areas. Before any UI, API, or workflow development can begin, the data layer must be defined and implemented. This issue covers: core `logistics_records` table, `logistics_details` table with status workflow (draft → submitted → approved), monthly partitioning (`logistics_records_y2026m04`), Redis caching configuration, audit trail event types, RLS policies for multi-org isolation, and seed data for development environments. No API or UI work is included.

| 1A | Create `logistics_records` table with UUID PK (autogenerated `01700-{YYYY}-{NNNNN}` format), RLS, composite indexes on (company_id, status, created_at) | Table created with all constraints |
